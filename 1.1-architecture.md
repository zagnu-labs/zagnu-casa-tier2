# Architecture Overview

This document describes the high-level architecture of the Zagnu platform — its components, trust boundaries, third-party integrations, and the main data flows for sensitive operations (notably the Gmail OAuth integration).

## 2. Components

### Frontend

A Next.js Progressive Web App hosted on Vercel. Serves the marketing site, onboarding flow, and the authenticated user app. All sensitive operations are delegated to the backend over HTTPS.

### Public API gateway

An AWS Application Load Balancer fronts every backend service. TLS is terminated at the gateway with an ACM-managed certificate. The gateway routes traffic to the correct microservice based on URL path prefix.

### Backend microservices

All backend services run as Go applications inside a single AWS EKS cluster, in a private VPC.

| Service | Responsibility |
|---|---|
| `account-registry` | User and account management |
| `payment-engine` | Payment processing and ledger |
| `zmail-services` | Reads connected Gmail inboxes to detect and attest payment confirmation emails; emits events for the rest of the platform |

### Data stores

- **PostgreSQL (Aurora)** — primary relational store. Each microservice owns its own tables; cross-service writes are not permitted.
- **Redis (ElastiCache)** — short-lived key/value store used for ephemeral data such as short-link codes (30-minute TTL).
- **Kafka (MSK)** — internal event bus. Communication is authenticated with SASL/SCRAM-SHA-512 and encrypted in transit with TLS. Services consume each other's events instead of making direct HTTP calls when possible.

## 3. Third-party integrations

| Vendor | Purpose | Data shared |
|---|---|---|
| Google (Gmail API) | Read the user's inbox after explicit OAuth consent, scope `gmail.readonly` | OAuth tokens issued to us; query filters; the user's email address |
| OpenAI | Structured extraction of payment fields from email bodies | Body of a payment-notification email |
| Vercel | Hosting of the frontend | Page requests and static assets only |
| AWS | All backend infrastructure (compute, networking, databases, event bus) | All customer data resides here |

No customer data is shared with any third party other than the four above.

---

## 4. Trust boundaries

The platform recognises the following trust boundaries:

1. **End user → Frontend.** The browser is untrusted. All input is treated as adversarial until validated server-side.
2. **Frontend → API gateway.** The gateway only accepts HTTPS. CORS is restricted to the production frontend origin.
3. **API gateway → Internal services.** Only the gateway is publicly reachable. Internal services live in a private subnet and never accept inbound traffic from the public internet.
4. **Services → Data stores.** Databases and Kafka are inside the cluster's VPC. Kafka enforces SASL/SCRAM authentication and TLS. Database credentials are injected as Kubernetes secrets at deploy time.
5. **Service → External third parties.** Only the explicitly listed third parties (Google, OpenAI, Vercel, AWS) are reachable. All connections are HTTPS.

## 5. Sensitive data handling

| Data | Where it lives | Encryption |
|---|---|---|
| Google OAuth access and refresh tokens | PostgreSQL, in the `zmail-services` schema | At-rest via Aurora encryption; in-transit via TLS |
| Email bodies (transient) | Sent to OpenAI per extraction request; never persisted | In-transit via TLS |
| Extracted payment data (amount, transaction id, sender, memo) | PostgreSQL | At-rest via Aurora |
| User email address | PostgreSQL | At-rest via Aurora |
| Short-link codes | Redis (30-minute TTL) | In-transit only; codes are opaque random strings with no PII |

OAuth tokens never leave the backend. Email bodies are sent to OpenAI exclusively for extraction; OpenAI's API policy does not allow this data to be used to train their models.

## 6. Gmail OAuth flow

The Gmail integration is the most security-sensitive flow on the platform. It uses the standard Google OAuth 2.0 authorization-code flow with the `gmail.readonly` scope.

1. The user requests to connect their inbox from the frontend.
2. The backend generates the Google consent URL and stores it behind a short-lived random code in Redis. A short link (e.g. `https://zagnu.com/t/<code>`) is returned to the user — this is what is delivered over SMS.
3. The user opens the short link. The frontend rewrites it to the backend's resolve endpoint, which 302-redirects to the Google consent screen.
4. On consent, Google redirects the user to the backend's OAuth callback. The backend exchanges the authorization code for tokens, queries the user's email address from Google, and persists the tokens in PostgreSQL.
5. The user is redirected back to the frontend's onboarding page (the destination URL is configurable per environment).
6. Subsequent inbox reads use the stored refresh token to mint short-lived access tokens. Tokens are refreshed automatically by the OAuth client library; refreshed tokens are written back to PostgreSQL.

The tokens are used only for read operations against the connected user's Gmail inbox. The platform never sends, modifies, or deletes mail.

---

## 7. Authentication and authorization

- **End-user authentication** at the platform level is performed at the `account-registry` boundary. Authenticated session credentials are required for every request that accesses user-scoped data.
- **Gmail integration authorization** is performed by Google through its OAuth 2.0 consent screen. The user can revoke access at any time from their Google account settings; the backend handles revocation events by invalidating the corresponding session.
- **Administrative endpoints** are not exposed on the public internet.

---

## 8. Infrastructure and change management

- Infrastructure is declared as code (Terraform for AWS resources, Helm for Kubernetes deployments).
- Deployments are performed by CI on push to the main branch. Container images are built and stored in AWS ECR. Secrets are managed centrally and injected into Kubernetes at deploy time.
- The frontend deploys via Vercel's GitHub integration on push to the main branch.
